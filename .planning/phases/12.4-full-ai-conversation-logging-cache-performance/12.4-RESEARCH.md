## Summary

This phase has two independent tracks. Track 1 fixes AI conversation logging: today the route only logs when a tool fires (`suggest_genres` or `identify_media` execute blocks), using synthetic assistant messages (`"Suggested: Comedy, Drama"` strings). The fix is to move all logging into `streamText`'s `onFinish` callback, which fires after every complete response — capturing the actual streamed text verbatim, regardless of whether a tool ran. Track 2 fixes page-navigation loading: /discover and /series are pure Server Components that pass data directly to client components via props today, but those client components call TanStack Query hooks that re-fetch on mount because nothing has seeded the cache. The fix is to add `prefetchQuery` + `HydrationBoundary` in the Server Components so the browser TanStack Query cache is pre-seeded from the SSR data.

Both tracks are well-understood. The codebase already has all the patterns needed. The AI SDK v6 (installed: `^6.0.82`) exposes `onFinish` on `streamText` with an `event.text` field containing the final aggregated text and `event.steps[]` for per-step data. TanStack Query v5's `dehydrate` + `HydrationBoundary` is the standard SSR pattern for App Router.

**Primary recommendation:** Add `onFinish` to the single `streamText` call to consolidate all conversation logging there, and add `prefetchQuery`+`HydrationBoundary` to the /discover and /series Server Components, mirroring what Home does.

## Architecture Patterns

### Pattern 1: AI SDK v6 `onFinish` Callback

**What:** `streamText` accepts an `onFinish` callback that fires once, after all steps complete. It receives an `OnFinishEvent` with `.text` (aggregated final text across all steps) and `.steps[]` (per-step details including tool calls).

**When to use:** Consolidate all AI conversation logging here instead of per-tool execute blocks. This captures every response — whether tools ran or not.

**Signature (verified from installed `ai` v6 type definitions):**
```typescript
// Source: node_modules/ai/dist/index.d.ts
type OnFinishEvent<TOOLS extends ToolSet> = StepResult<TOOLS> & {
  readonly steps: StepResult<TOOLS>[];
  readonly totalUsage: LanguageModelUsage;
};

// StepResult has:
//   readonly text: string           — the AI's final text
//   readonly content: ContentPart[] — includes tool-call parts
//   readonly toolCalls: ...         — tool calls made in last step
//   readonly toolResults: ...       — results from last step

streamText({
  // ...
  onFinish: async (event) => {
    // event.text — actual AI response text (verbatim, all steps aggregated)
    // event.steps — all steps; inspect for tool calls
    // event.steps[*].toolCalls — [{toolName, args}]
    // event.steps[*].toolResults — [{toolName, result}]
  }
});
```

**Example — consolidated logging in `onFinish`:**
```typescript
// In app/api/ai/recommend/route.ts
const result = streamText({
  model: geminiModel,
  system: systemPrompt,
  messages: modelMessages,
  tools: { suggest_genres: ..., identify_media: ... },
  stopWhen: stepCountIs(3),
  maxOutputTokens: 1000,
  maxRetries: 0,
  onFinish: (event) => {
    // Derive metadata from steps
    let toolUsed: "suggest_genres" | "identify_media" | "none" = "none";
    let genresSuggested: string[] | undefined;
    let mediaIdentified: string[] | undefined;

    for (const step of event.steps) {
      for (const tc of step.toolCalls ?? []) {
        if (tc.toolName === "suggest_genres") {
          toolUsed = "suggest_genres";
          const args = tc.args as { genres?: Array<{ name: string }> };
          genresSuggested = args.genres?.map((g) => g.name);
        } else if (tc.toolName === "identify_media") {
          toolUsed = "identify_media";
        }
      }
      for (const tr of step.toolResults ?? []) {
        if (tr.toolName === "identify_media") {
          const res = tr.result as { matches?: Array<{ title: string }> };
          mediaIdentified = res.matches?.map((m) => m.title);
        }
      }
    }

    const metadata = {
      toolUsed,
      genresSuggested,
      mediaIdentified,
      messageCount: uiMessages.length + 1, // +1 for this AI response
      isOffTopic: false,
    };

    // Build full conversation: existing messages + new AI response
    const assistantMessage = {
      role: "assistant",
      parts: [{ type: "text", text: event.text }],
    };
    const fullMessages = [...uiMessages, assistantMessage];
    const moodPrompt = lastMessageText || "mood chat";

    if (conversationId) {
      db.insert(aiConversations)
        .values({ userId, prompt: moodPrompt, messages: fullMessages, conversationId, metadata })
        .onConflictDoUpdate({
          target: [aiConversations.conversationId],
          set: { messages: fullMessages, prompt: moodPrompt, updatedAt: new Date(), metadata },
        })
        .catch(() => {});
    } else {
      db.insert(aiConversations)
        .values({ userId, prompt: moodPrompt, messages: fullMessages, metadata })
        .catch(() => {});
    }
  },
});
```

**Key insight:** `onFinish` fires AFTER the stream completes and AFTER the response has already been returned to the client. It is inherently async/non-blocking with respect to the HTTP response. This means fire-and-forget is natural here — no `.catch()` trick needed for latency, though still good practice.

**Off-topic path:** The off-topic early-return path uses `createUIMessageStreamResponse` (not `streamText`), so `onFinish` does NOT fire for it. Off-topic conversations that exit early must be logged separately before returning the stream response. A simple fire-and-forget insert before the `return createUIMessageStreamResponse(...)` call handles this.

### Pattern 2: TanStack Query SSR Hydration (App Router)

**What:** Server Components call `prefetchQuery` on a server-side `QueryClient`, then pass the dehydrated state to `HydrationBoundary`. Client components in the subtree find their queries already populated — no loading state on first render or on return visits (stale-while-revalidate kicks in).

**When to use:** Any page where a Server Component already fetches data that client-side TanStack Query hooks would otherwise re-fetch on mount.

**Confirmed pattern (TanStack Query v5 official docs + verified in installed codebase):**

```typescript
// In a Server Component (e.g. app/(app)/discover/page.tsx)
import { dehydrate, HydrationBoundary, QueryClient } from "@tanstack/react-query";
import { getCachedTrending, getCachedPopular, getCachedTopRated } from "@/lib/tmdb-cache";

export default async function DiscoverPage() {
  const queryClient = new QueryClient();

  await Promise.all([
    queryClient.prefetchQuery({
      queryKey: movieKeys.category("trending", 1),
      queryFn: () => getCachedTrending(1),
    }),
    queryClient.prefetchQuery({
      queryKey: movieKeys.category("popular", 1),
      queryFn: () => getCachedPopular(1),
    }),
    queryClient.prefetchQuery({
      queryKey: movieKeys.category("top_rated", 1),
      queryFn: () => getCachedTopRated(1),
    }),
  ]);

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <DiscoverGridContent />  {/* unchanged client component */}
    </HydrationBoundary>
  );
}
```

**Key behaviour:** `HydrationBoundary` merges the server-dehydrated cache into the browser `QueryClient` (which is the singleton `browserQueryClient` in `providers.tsx`). On first visit, client components get data immediately. On return visits within `staleTime`, they get cached data instantly. After `staleTime`, they show cached data AND silently revalidate — no skeleton flash.

**Important:** `makeQueryClient()` must NOT be called in Server Components — use `new QueryClient()` directly (or a fresh factory call) to avoid the browser singleton. The `makeQueryClient()` in `providers.tsx` is for the browser singleton only.

**Providers.tsx compatibility:** The existing `Providers` component is already compatible with `HydrationBoundary`. No changes needed to `providers.tsx`.

## Common Pitfalls

### Pitfall 1: Off-topic Path Bypasses `onFinish`
**What goes wrong:** The off-topic early-return uses `createUIMessageStream` / `createUIMessageStreamResponse` — not `streamText`. The `onFinish` callback never fires for these paths.
**Why it happens:** `onFinish` is a property of `streamText`'s result, not of the transport stream.
**How to avoid:** Add a separate fire-and-forget log insert before the early return for off-topic. Include `isOffTopic: true` in metadata. The insert is a single simple write with no upsert needed (no conversationId update).
**Warning signs:** Off-topic conversations show up with 0 DB entries — check by querying `WHERE metadata->>'isOffTopic' = 'true'` in Drizzle Studio.

### Pitfall 2: Removing Tool Execute Logging Before `onFinish` Is Verified
**What goes wrong:** Tool execute blocks currently do the logging. If they are deleted before `onFinish` is tested, there's a gap with no logging.
**How to avoid:** Add `onFinish` logging first, verify it works (spot-check a conversation in Drizzle Studio), then remove the execute-block logging in a follow-up commit.

### Pitfall 3: `HydrationBoundary` State Not Reaching Browser Client
**What goes wrong:** `prefetchQuery` runs in a Server Component but the dehydrated state isn't passed down — data still fetches on mount. Usually caused by forgetting to wrap the client subtree in `<HydrationBoundary state={dehydrate(queryClient)}>`.
**How to avoid:** The `state` prop must be `dehydrate(queryClient)` not just `queryClient`. Always call `dehydrate()` explicitly.
**Warning signs:** React Query DevTools shows the query re-fetching on mount with `fetchStatus: "fetching"` and no initial data.

### Pitfall 4: Query Key Mismatch Between `prefetchQuery` and Hook
**What goes wrong:** Server-side `prefetchQuery` uses `movieKeys.category("trending", 1)` but client hook uses a different key shape. Cache miss occurs; skeleton still shows.
**How to avoid:** Use the exact same `movieKeys.*` factory functions in both `prefetchQuery` and the hook. These are exported from `hooks/use-movies.ts`. Import them directly in Server Components.
**Note:** `hooks/use-movies.ts` has `"use client"` at the top but the key factories themselves are plain objects/functions — they don't use React hooks. It is safe to import `movieKeys` and `tvKeys` in Server Components.

### Pitfall 5: `makeQueryClient()` Singleton Leak in Server Components
**What goes wrong:** If `makeQueryClient()` (from `lib/query-client.ts`) is called in a Server Component, its singleton check (`if (!browserQueryClient)`) doesn't apply — `window` is undefined — so a fresh client is made. This is fine. BUT if the Server Component accidentally uses the browser singleton, different requests could share cache.
**How to avoid:** Always `new QueryClient()` (inline) in Server Components for `prefetchQuery`. Never pass the `browserQueryClient` into a Server Component.

### Pitfall 6: `metadata` Column Typing in Drizzle Insert
**What goes wrong:** TypeScript complains about the JSONB `metadata` field type — `unknown` vs `Record<string, unknown>`.
**How to avoid:** Cast the metadata object explicitly: `metadata: metadata as Record<string, unknown>` in the insert `.values()` call. This matches the pattern already used for `messages: fullMessages` in the existing logging code.

## Current State Analysis

### AI Conversation Logging — What Exists Today

The current `route.ts` has two separate logging blocks — one inside `suggest_genres.execute` and one inside `identify_media.execute`. Both write synthetic assistant messages:
- `suggest_genres`: `"Suggested: Comedy, Drama"` — not the actual AI text
- `identify_media`: `"Identified: The Martian, ..."` — not the actual AI text
- Off-topic path: **no logging at all**
- Clarifying Q&A (no tool fires): **no logging at all**

The `onFinish` migration addresses all four gaps in a single callback.

### TanStack Query Cache — What Exists Today

- `/home`: Server Component fetches data via `getCachedTrending` etc. and passes as **React props** to `HomeMovies`. The props are used directly in renders. TanStack Query cache is NOT seeded. On return visits, `HomeMovies` client components do re-fetch via hooks BUT `staleTime: 10 * 60 * 1000` (5 min default) means return visits within 10 min show stale data immediately.
- `/discover`: Server Component passes nothing. `DiscoverGridContent` is pure client. Every visit triggers fresh fetches.
- `/series`: Server Component passes props directly. `SeriesPageContent` receives props. Does NOT use TanStack Query at all for the initial rows — no re-fetch issue for those. However the page re-SSRs on every navigation back to it (Next.js App Router caches SSR for `dynamic = 'force-static'` only; default is per-request).
- `/library`: Pure client. `WatchlistContent` calls `useWatchlist`. TanStack Query default `staleTime: 5min` handles this adequately.

**Key insight:** For /discover, adding `prefetchQuery` + `HydrationBoundary` will eliminate the initial skeleton AND give stale-while-revalidate on returns. For /series, the existing SSR-props pattern already works for initial load; the improvement is seeding TanStack Query cache so the movie rows have stale-while-revalidate on return.

## Open Questions

1. **`DiscoverGridContent` hook alignment with `prefetchQuery` keys**
   - What we know: `DiscoverGridContent` uses `useDiscoverMovies` (infinite query) and category hooks. The category hooks use `movieKeys.category(cat, page)` which matches the proposed `prefetchQuery` keys.
   - What's unclear: Whether `DiscoverGridContent` starts with the default sort/filter (no genre, no year) or a user-persisted filter state. If it starts with custom params, the prefetched key for the default view won't match.
   - Recommendation: Prefetch the default view (trending, popular, top_rated page 1) — these are always shown on first render before any filter is applied.

2. **`onFinish` with `stopWhen: stepCountIs(3)` — which step count's text is `event.text`?**
   - What we know: `event.text` is documented as the final step's text aggregated. With multi-step tool calls, the final conversational text is in the last text-producing step.
   - What's unclear: If step 1 = text, step 2 = tool call, step 3 = tool result + follow-up text — does `event.text` capture the final conversational response?
   - Recommendation: Test with a real conversation. The AI SDK docs say `text` aggregates all text across steps. Verify in Drizzle Studio that logged messages contain readable AI personality text (not just the tool execution text).
