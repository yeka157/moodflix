# Phase 12.4: Full AI Conversation Logging & Cache Performance - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Two improvements:
1. **Full AI conversation logging** — Log every user message and every AI response (actual streamed text) to the database, for all conversations including clarifying Q&A and off-topic redirects. Backend analytics only, no user-facing history.
2. **Cache performance** — Eliminate loading skeletons when navigating between app pages. SSR prefetch for first visits, instant cached content for return visits via stale-while-revalidate.

</domain>

<decisions>
## Implementation Decisions

### AI Conversation Logging Scope
- Log ALL conversations — every user prompt and every AI response, not just when tools fire
- Includes clarifying Q&A that never produces a recommendation
- Includes off-topic redirects (useful for analytics on what users actually ask)
- Save after every AI response (upsert full conversation) — no data loss if user closes tab
- Capture the AI's actual streamed text verbatim (not synthetic summaries like "Suggested: Comedy, Drama")
- Use `onFinish`/`onStepFinish` callback on `streamText` to capture final text after streaming completes
- Fire-and-forget pattern preserved — never add latency to streaming response
- Upsert by `conversationId` (existing pattern from Phase 12.1)

### Conversation Metadata
- Add a `metadata` JSONB column to `ai_conversations` table
- Store structured data: tool type used (suggest_genres/identify_media/none), genres suggested, media identified, message count, whether conversation was off-topic
- Makes Drizzle Studio queries easier for analytics

### Conversation Viewing
- Backend analytics only — no user-facing conversation history
- Drizzle Studio is sufficient for browsing logged conversations
- No admin UI needed
- Users always start fresh chats (no conversation continuity)

### Page Navigation Loading UX
- Goal: instant cached content when returning to any previously-visited page (stale-while-revalidate)
- No skeleton, no flash — show cached data immediately, silently revalidate in background
- Add SSR prefetching (TanStack Query `prefetchQuery` + `HydrationBoundary`) for first visits to /discover and /series
- This eliminates the initial loading skeleton on first visit too — consistent with home page behavior
- Applies to all app pages: Home, Discover, Series, Library

### Claude's Discretion
- Whether Library page needs SSR prefetching (evaluate if Supabase latency warrants it)
- TanStack Query `gcTime` and `staleTime` tuning values
- How to structure the `HydrationBoundary` wrapper (per-page vs shared layout)
- Implementation of `onFinish` callback for capturing streamed text
- Metadata column schema design (which fields to extract)

</decisions>

<specifics>
## Specific Ideas

- User noticed multi-second loading skeletons when navigating back to pages — "I thought because it's cached, since I'm using TanStack Query as well, then I can minimize the loading state time"
- The root cause: /discover and /series are pure client-side components that fire TanStack Query on mount. No `dehydrate`/`HydrationBoundary`/`prefetchQuery` anywhere in the codebase. Every navigation triggers fresh client-side fetches.
- Home page is SSR (passes props directly) but doesn't hydrate TanStack Query cache, so client components on home still re-fetch
- Current AI logging only captures conversations at tool execution time, with synthetic assistant messages — misses the actual AI personality and clarifying exchanges

</specifics>

- Fire-and-forget: `db.insert(...).catch(() => {})` — preserve for all logging
- SSR data passing: Home page fetches in Server Component and passes as props to client components
- TanStack Query key factory: `movieKeys.trending()`, `tvKeys.trending()`, etc.
- `conversationId` upsert: partial unique index, `onConflictDoUpdate` pattern

### Integration Points
- SSR prefetch: `app/(app)/discover/page.tsx`, `app/(app)/series/page.tsx` — add `prefetchQuery` + `HydrationBoundary`
- AI logging: `app/api/ai/recommend/route.ts` — add `onFinish` for text capture, update off-topic path to also log
- Schema: `drizzle/schema.ts` — add `metadata` JSONB column to `aiConversations`
- Provider: `components/providers.tsx` — may need QueryClient dehydration support
