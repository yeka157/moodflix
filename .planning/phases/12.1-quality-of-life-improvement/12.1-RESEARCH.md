### Pattern 2: Bottom Tab Bar Active State on Detail Pages

**What:** Detail routes `/movie/[id]` and `/tv/[id]` don't match any tab's `startsWith()` check, so no tab highlights.

**File:** `components/layout/bottom-tab-bar.tsx`

**Current logic:**
```typescript
function isActive(pathname: string, href: string, exact: boolean) {
  if (exact) return pathname === href;
  return pathname.startsWith(href);
}
```

**Options (Claude's discretion):**

1. **Dim all tabs (recommended):** Show all tabs in muted state. Detail pages are contextually independent -- the user could have arrived from /home, /discover, /series, or /library. Highlighting a "parent" tab would be misleading since there's no guaranteed parent. This is the pattern Netflix and Disney+ use on mobile.

2. **Track referrer tab:** Store which tab the user came from in a query param or context. More complex, fragile with browser back/forward, and provides marginal UX benefit.

**Recommendation:** Option 1 -- no active highlight on detail pages. The current behavior (no highlight) is actually correct; the issue is that the bar exists but looks "broken." Add subtle visual feedback that it's a detail page context: keep all tabs in muted-foreground (current default for inactive), which is already the behavior. The real fix is pairing this with the back button so users have clear navigation.

**Confidence:** HIGH

### Pattern 3: Back Navigation on Mobile Detail Pages

**What:** Mobile users on `/movie/[id]` or `/tv/[id]` have no way to go back -- swipe refreshes the page.

**Files:** The detail page client components that render the detail content.

**Implementation approach:**
```typescript
"use client";
import { useRouter } from "next/navigation";
import { ChevronLeft } from "lucide-react";

// Floating back button overlaid on the backdrop image
<button
  onClick={() => router.back()}
  className="fixed top-4 left-4 z-50 md:hidden size-10 rounded-full bg-black/60 backdrop-blur-sm flex items-center justify-center"
  aria-label="Go back"
>
  <ChevronLeft className="size-5 text-white" />

```

**Style recommendation (Claude's discretion):** Floating overlay button. The detail pages have a cinematic full-width backdrop image at the top. A header bar would break the immersive design. A floating semi-transparent circle (like Netflix/YouTube mobile) overlaid on the backdrop is the standard pattern for cinematic detail pages. Position: `fixed top-4 left-4 z-50`, show only on mobile (`md:hidden`), use `bg-black/60 backdrop-blur-sm` for readability over any backdrop image.

**Confidence:** HIGH -- standard mobile pattern, `useRouter().back()` is reliable.

### Pattern 4: Multi-Turn Conversation Logging

**What:** Current logging captures `uiMessages` at the time `suggest_genres` tool fires. In a multi-turn conversation, this only captures messages up to the first genre suggestion. Subsequent turns (user refines, AI suggests again) are lost.

**File:** `app/api/ai/recommend/route.ts` lines 262-271

**Current code:**
```typescript
db.insert(aiConversations)
  .values({
    userId,
    prompt: moodPrompt,
    messages: [...uiMessages, { role: "assistant", parts: [...] }],
  })
  .catch(() => {});
```

**Problem:** `uiMessages` comes from `parsed.data.messages` -- the full message array sent by the client on each request. For multi-turn, the client sends ALL accumulated messages each time (this is how `useChat` works). So `uiMessages` already contains the full conversation history on each request.

**The actual issue:** Each tool call creates a NEW row in `ai_conversations`. If the user has a 3-turn conversation with 2 genre suggestions, there will be 2 rows -- the second row contains the full conversation (superset of the first). This is redundant but not data-losing.

**Schema recommendation (Claude's discretion):** Keep the current schema unchanged. The `messages` JSONB column already stores the full array. The approach should be:
- On each `suggest_genres` call, upsert or simply insert the latest full conversation
- Alternatively, insert only once per "session" -- but there's no session concept in the current stateless API

**Simplest correct approach:** Continue inserting a new row each time, but accept that later rows supersede earlier ones for the same conversation. This is already how it works. If the concern is that "subsequent turns are lost" -- verify that `uiMessages` actually contains the full history on subsequent requests (it should, since `useChat` sends all messages).

If the user wants exactly one row per conversation, add a `conversationId` (UUID generated client-side, passed with each request) and use upsert:
```typescript
db.insert(aiConversations)
  .values({ userId, prompt: moodPrompt, messages: fullMessages, conversationId })
  .onConflictDoUpdate({
    target: [aiConversations.conversationId],
    set: { messages: fullMessages, prompt: moodPrompt },
  })
  .catch(() => {});
```

This requires a small schema addition (conversationId column + unique constraint) and client-side UUID generation in the `useMoodChat` hook.

**Confidence:** HIGH -- the AI SDK v5 `useChat` sends the full message array on every request, so the data is available.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Back navigation | Custom swipe gesture handler | `useRouter().back()` | Browser history handles edge cases |
| Tab state tracking | Context/localStorage for "came from" tab | No highlight on detail pages | Simpler, less fragile, matches industry pattern |
| Conversation dedup | Manual dedup logic | Upsert with conversationId OR accept multiple rows | DB handles atomicity |

## Common Pitfalls

### Pitfall 1: router.back() with no history
**What goes wrong:** If user lands directly on `/movie/123` (shared link), `router.back()` navigates away from the app.
**How to avoid:** Fallback to `router.push("/home")` when there's no history. Check `window.history.length <= 1` or use `document.referrer`.

### Pitfall 2: Fixed positioning + bottom tab bar overlap
**What goes wrong:** Fixed back button at top is fine, but page content must still account for the 64px bottom tab bar on mobile.
**How to avoid:** Detail pages likely already have `pb-16 md:pb-0` -- verify this.

### Pitfall 3: Send button accessibility
**What goes wrong:** Changing the icon without updating the tooltip/aria-label.
**How to avoid:** The tooltip is already "Send message" (line 276) -- just verify it stays correct.

### Pitfall 4: Conversation logging race condition
**What goes wrong:** Two rapid tool calls could insert in unexpected order.
**How to avoid:** Not a real problem with fire-and-forget inserts -- each row is independent. If using upsert approach, the last write wins, which is the correct behavior.

## Code Examples

### Back Button Component (mobile only)
```typescript
"use client";
import { useRouter } from "next/navigation";
import { ChevronLeft } from "lucide-react";

export function MobileBackButton() {
  const router = useRouter();

  const handleBack = () => {
    if (window.history.length > 1) {
      router.back();
    } else {
      router.push("/home");
    }
  };

  return (
    <button
      onClick={handleBack}
      className="fixed top-4 left-4 z-50 md:hidden size-10 rounded-full bg-black/60 backdrop-blur-sm flex items-center justify-center transition-colors active:bg-black/80"
      aria-label="Go back"
    >
      <ChevronLeft className="size-5 text-white" />
    </button>
  );
}
```
