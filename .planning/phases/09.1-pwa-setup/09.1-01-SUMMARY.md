## Decisions Made
- Used `@serwist/turbopack` (not webpack mode) to avoid conflict with Sentry's `withSentryConfig` webpack plugin in next.config.ts
- Set `theme_color: "#09090b"` (dark background) per discretion recommendation — crimson accent looks off as browser chrome color
- Excluded `app/sw.ts` from tsconfig.json `exclude` array — triple-slash `no-default-lib` directive was polluting project-wide TS type resolution
- Created thin `components/pwa/sw-register.tsx` client wrapper — SerwistProvider cannot be directly imported in a Server Component layout without "use client" boundary

**1. [Rule 1 - Bug] Excluded app/sw.ts from tsconfig.json to fix webworker type conflicts**
- **Found during:** Task 2 (build verification)
- **Issue:** app/sw.ts triple-slash `/// <reference no-default-lib="true" />` directive caused TypeScript to lose DOM lib types for other files. This broke `scrollTop`/`scrollHeight` property access in `components/ai/mood-section.tsx`, failing the production build
- **Fix:** Added `"app/sw.ts"` to the `exclude` array in tsconfig.json — standard pattern per Serwist documentation for webworker files

## Issues Encountered
- Build initially failed because `app/sw.ts` uses triple-slash `no-default-lib` which when compiled alongside regular DOM files caused TypeScript to lose HTMLElement DOM type definitions project-wide. Resolved by excluding the service worker source from the main tsconfig compilation scope — this is the documented Serwist approach for Turbopack mode.
