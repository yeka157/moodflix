## Summary

Moodflix already has a partial PWA foundation: a `manifest.ts` in `app/`, a redundant `public/site.webmanifest`, basic icon PNGs (`icon-192.png`, `icon-512.png`, `icon-maskable.png`), and proper meta linkage in `layout.tsx`. However, the manifest link in layout.tsx still points to `/site.webmanifest` (the old static file) instead of the App Router's `app/manifest.ts` route (`/manifest.webmanifest`). There is no service worker, no offline page, and no install prompt. The app cannot be installed as a PWA today.

The recommended service worker approach for Next.js 16 is **Serwist** — officially mentioned in the Next.js PWA docs and the only library with active Turbopack support. Since Moodflix uses Next.js 16 (Turbopack is the default), the correct package is `@serwist/turbopack` (not the webpack-based `@serwist/next`). Sentry already uses the webpack plugin path; using `@serwist/turbopack` avoids bundler conflicts entirely, since the Turbopack path uses a Next.js route handler instead of a webpack plugin. The icon set already exists but needs redesigning (cinematic crimson/dark treatment per locked decisions) and a maskable safe-zone check.

**Primary recommendation:** Use `@serwist/turbopack` + `serwist` for service worker/caching, unify the manifest to `app/manifest.ts` only, generate new cinematic icons with `sharp` or a PNG editor, add `app/~offline/page.tsx` as the offline fallback, and add a `useOnlineStatus` hook wired to `sonner` for connectivity toasts.

### Pattern 1: Turbopack Route Handler (Service Worker Generation)

**What:** Instead of a webpack plugin that modifies the build, `@serwist/turbopack` exposes a Next.js App Router route handler at `app/serwist/[path]/route.ts`. On first visit, the browser fetches `/serwist/sw.js` (served by this handler), which is the compiled service worker.

**When to use:** Always — this is the only approach that works with Turbopack + Sentry simultaneously.

```typescript
// app/serwist/[path]/route.ts
// Source: https://serwist.pages.dev/docs/next/turbo
import { spawnSync } from "node:child_process";
import { createSerwistRoute } from "@serwist/turbopack";

const revision =
  spawnSync("git", ["rev-parse", "HEAD"], { encoding: "utf-8" }).stdout ??
  crypto.randomUUID();

export const {
  dynamic,
  dynamicParams,
  revalidate,
  generateStaticParams,
  GET,
} = createSerwistRoute({
  additionalPrecacheEntries: [{ url: "/~offline", revision }],
  swSrc: "app/sw.ts",
  useNativeEsbuild: true,
});
```

### Pattern 2: Service Worker with Runtime Caching

**What:** The `app/sw.ts` file defines precaching (app shell via `__SW_MANIFEST`) and runtime caching rules for images and API responses.

```typescript
// app/sw.ts
// Source: https://serwist.pages.dev/docs/next/turbo
/// <reference no-default-lib="true" />
/// <reference lib="esnext" />
/// <reference lib="webworker" />

import { defaultCache } from "@serwist/turbopack/worker";
import type { PrecacheEntry, SerwistGlobalConfig } from "serwist";
import {
  CacheFirst,
  CacheableResponsePlugin,
  ExpirationPlugin,
  NetworkFirst,
  StaleWhileRevalidate,
  Serwist,
} from "serwist";

declare global {
  interface WorkerGlobalScope extends SerwistGlobalConfig {
    __SW_MANIFEST: (PrecacheEntry | string)[] | undefined;
  }
}
declare const self: ServiceWorkerGlobalScope;

const serwist = new Serwist({
  precacheEntries: self.__SW_MANIFEST,
  skipWaiting: true,
  clientsClaim: true,
  navigationPreload: true,
  runtimeCaching: [
    // TMDB images — CacheFirst, cache only what user has seen
    {
      matcher: ({ request, url }) =>
        request.destination === "image" &&
        url.hostname === "image.tmdb.org",
      handler: new CacheFirst({
        cacheName: "tmdb-images",
        plugins: [
          new CacheableResponsePlugin({ statuses: [0, 200] }),
          new ExpirationPlugin({
            maxEntries: 500,       // ~50-100MB range at ~200KB avg per poster
            maxAgeSeconds: 30 * 24 * 60 * 60, // 30 days
          }),
        ],
      }),
    },
    // Own API routes — NetworkFirst with cache fallback
    {
      matcher: ({ url }) =>
        url.pathname.startsWith("/api/movies") ||
        url.pathname.startsWith("/api/ai"),
      handler: new NetworkFirst({
        cacheName: "api-routes",
        plugins: [
          new CacheableResponsePlugin({ statuses: [200] }),
          new ExpirationPlugin({
            maxEntries: 50,
            maxAgeSeconds: 5 * 60, // 5 minutes
          }),
        ],
      }),
    },
    // Spread the Next.js default cache on top
    ...defaultCache,
  ],
  fallbacks: {
    entries: [
      {
        url: "/~offline",
        matcher({ request }) {
          return request.destination === "document";
        },
      },
    ],
  },
});

serwist.addEventListeners();
```

### Pattern 4: useOnlineStatus Hook

**What:** Hook using `navigator.onLine` + `window` event listeners for `online`/`offline` events. Wire to Sonner toasts.

```typescript
// hooks/use-online-status.ts
"use client";
import { useEffect, useState } from "react";

export function useOnlineStatus() {
  const [isOnline, setIsOnline] = useState(
    typeof navigator !== "undefined" ? navigator.onLine : true
  );

  useEffect(() => {
    const handleOnline = () => setIsOnline(true);
    const handleOffline = () => setIsOnline(false);
    window.addEventListener("online", handleOnline);
    window.addEventListener("offline", handleOffline);
    return () => {
      window.removeEventListener("online", handleOnline);
      window.removeEventListener("offline", handleOffline);
    };
  }, []);

  return isOnline;
}
```

### Pattern 5: Install Prompt (Chrome/Android only — discretion)

**What:** `beforeinstallprompt` fires on Chromium browsers when PWA criteria are met. iOS requires manual "Add to Home Screen" via Share. **Note from Next.js docs:** They explicitly recommend against custom install prompts because they don't work cross-browser. Recommendation: show a subtle sticky banner only on Chrome/Android using `beforeinstallprompt` detection; iOS users get no automatic prompt (they must use Safari Share sheet).

```typescript
// hooks/use-pwa-install.ts — discretion item
"use client";
import { useEffect, useState } from "react";

interface BeforeInstallPromptEvent extends Event {
  prompt(): Promise<void>;
  userChoice: Promise<{ outcome: "accepted" | "dismissed" }>;
}

export function usePwaInstall() {
  const [deferredPrompt, setDeferredPrompt] =
    useState<BeforeInstallPromptEvent | null>(null);

  useEffect(() => {
    const handler = (e: Event) => {
      e.preventDefault();
      setDeferredPrompt(e as BeforeInstallPromptEvent);
    };
    window.addEventListener("beforeinstallprompt", handler);
    return () => window.removeEventListener("beforeinstallprompt", handler);
  }, []);

  const install = async () => {
    if (!deferredPrompt) return;
    await deferredPrompt.prompt();
    setDeferredPrompt(null);
  };

  return { canInstall: !!deferredPrompt, install };
}
```

### Pattern 6: Manifest Cleanup

**What:** Moodflix has two manifests today. The `app/manifest.ts` App Router route serves at `/manifest.webmanifest`, but `layout.tsx` links to `/site.webmanifest` (the static file). Fix: delete `public/site.webmanifest`, update `layout.tsx` `manifest` metadata field to use the App Router route (or remove the explicit override so Next.js auto-links `app/manifest.ts`).

**CRITICAL:** The current `app/manifest.ts` points to `/icon-192.png` etc. Update icon paths to the new redesigned icons and add the correct maskable entry.

### Anti-Patterns to Avoid

- **Using `@serwist/next` (webpack mode) with Sentry:** Sentry already configures webpack in `next.config.ts` via `withSentryConfig`. Adding another webpack plugin via `@serwist/next` can cause "Webpack is configured while Turbopack is not" warnings and bundler conflicts. Use `@serwist/turbopack` instead.
- **Adding `webworker` to main `tsconfig.json` lib:** The Serwist Turbopack approach uses triple-slash directives (`/// <reference lib="webworker" />`) in `app/sw.ts` instead — this avoids polluting the main TypeScript scope with Worker globals. The webpack setup's tsconfig instruction (`"lib": ["webworker"]`) does NOT apply to the Turbopack setup.
- **Keeping `public/site.webmanifest`:** Two manifests will confuse browsers. The App Router `manifest.ts` is the canonical one.
- **Enabling service worker in development:** Always set service worker to disabled in dev (`disable: process.env.NODE_ENV === "development"` or equivalent). Service worker caching in dev causes stale content bugs.
- **Pre-fetching all TMDB images:** Cache only what users have actually viewed (runtime cache `CacheFirst`), never pre-fetch the entire TMDB catalog.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Service worker precaching | Custom SW with fetch event | Serwist `Serwist` class + `__SW_MANIFEST` | Cache invalidation on deploy, hash-based versioning, cleanup of old caches — all edge cases Workbox solves |
| Cache size enforcement | Manual cache size tracking | `ExpirationPlugin` with `maxEntries` + `maxAgeSeconds` | LRU eviction, age-based expiry, automatic cleanup on cache access — complex to get right |
| Offline page routing | Manual fetch event routing | Serwist `fallbacks.entries` with document matcher | Handles navigation requests vs subresource requests correctly |
| Service worker registration | Manual `navigator.serviceWorker.register()` | `SerwistProvider` from `@serwist/turbopack/react` | Handles update detection, reload-on-update prompting, error handling |

**Key insight:** Workbox (which Serwist wraps) has 7+ years of edge case handling for service worker caching. Cache versioning, storage quota handling, and opaque response caching (status 0) are all handled — never try to replicate this from scratch.

## Common Pitfalls

### Pitfall 1: Turbopack vs Webpack — Wrong Package

**What goes wrong:** Using `@serwist/next` (webpack mode) with a Next.js 16 project that uses Turbopack produces build errors or silent SW registration failures.

**Why it happens:** Serwist's webpack plugin injects into the webpack compilation. Turbopack does not support webpack plugins.

**How to avoid:** Always use `@serwist/turbopack` when the project uses Next.js 16 with default Turbopack. Check `next.config.ts` — if it wraps with `withSentryConfig`, it's already in webpack plugin territory; `@serwist/turbopack` side-steps this by using route handlers.

**Warning signs:** Build warns "Webpack is configured while Turbopack is not" or service worker file is never generated.

### Pitfall 2: Sentry + Webpack Plugin Conflict

**What goes wrong:** `withSentryConfig` wraps `next.config.ts` with webpack configuration. Adding `withSerwistInit` from `@serwist/next` adds a second webpack plugin, potentially causing "multiple webpack configurations" warnings in Next.js 15.3+.

**Why it happens:** Both Sentry and `@serwist/next` inject into the webpack compiler lifecycle.

**How to avoid:** Use `@serwist/turbopack` — it doesn't touch webpack at all. The composition is: `withSentryConfig(nextConfig, ...)` stays unchanged; `@serwist/turbopack` adds its own route handler separately.

**Warning signs:** `NextJs shows warning after adding Sentry: "Webpack is configured while Turbopack is not"`.

### Pitfall 3: Stale Service Worker in Development

**What goes wrong:** Cached assets from a previous dev session are served instead of fresh code. Very difficult to debug, especially with hot reload.

**Why it happens:** Service workers intercept fetch requests regardless of environment. Cache entries from yesterday's session are still served today.

**How to avoid:** The `createSerwistRoute` call should only be active in production. Check Serwist docs for environment-based disable flag, or wrap the route handler export with a `NODE_ENV !== "production"` guard that returns 404.

**Warning signs:** Changes to components not reflected in browser even after hard refresh.

### Pitfall 4: Two Manifests — Incorrect Manifest Link

**What goes wrong:** Browser uses `/site.webmanifest` (old, missing maskable icon) instead of `app/manifest.ts`. Install criteria fail silently or icons look bad.

**Why it happens:** `layout.tsx` currently sets `manifest: "/site.webmanifest"` explicitly. The App Router `manifest.ts` serves at `/manifest.webmanifest` (different URL). Both files are valid JSON so no error is thrown.

**How to avoid:** In the unified manifest task: delete `public/site.webmanifest`, update `layout.tsx` to remove the explicit `manifest` metadata override (Next.js will auto-link `app/manifest.ts`), or update it to `manifest: "/manifest.webmanifest"`.

**Warning signs:** Chrome DevTools > Application > Manifest shows the old icon paths or `theme_color: "#e11d48"` (old value) instead of the new crimson OKLCH value.

### Pitfall 5: iOS Splash Screen / Status Bar

**What goes wrong:** On iOS, the PWA splash screen uses the `background_color` from the manifest but the status bar color may conflict with the dark UI.

**Why it happens:** iOS PWA splash is auto-generated from manifest `background_color` + the largest icon. iOS does not support the `theme-color` meta for status bar in the same way Android does.

**How to avoid:** Ensure `background_color: "#09090b"` (Moodflix dark background) is set in manifest. Add `<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">` in `layout.tsx` for iOS status bar. Add `<meta name="apple-mobile-web-app-capable" content="yes">` for iOS installability.

**Warning signs:** White flash on launch, or status bar shows white on dark background.

### Pitfall 6: `beforeinstallprompt` — Not Cross-Platform

**What goes wrong:** Implementing an install button that only works on Chrome desktop/Android, breaking on iOS/Safari/Firefox with no fallback.

**Why it happens:** `beforeinstallprompt` is a Chromium-only non-standard event. It never fires on iOS Safari.

**How to avoid:** Build install prompt as progressive enhancement — only show the button when `deferredPrompt` is non-null (Chrome/Android only). For iOS, optionally show a static instruction: "Tap Share → Add to Home Screen". Never gate functionality on install status.

### Pitfall 7: TypeScript Conflicts with `webworker` lib

**What goes wrong:** Adding `"webworker"` to `tsconfig.json` lib array causes type conflicts between `ServiceWorkerGlobalScope` and `Window` in regular component files.

**Why it happens:** `webworker` adds `self` typed as `ServiceWorkerGlobalScope` globally, conflicting with `window.self` typed as `Window`.

**How to avoid:** The `@serwist/turbopack` approach uses triple-slash directives in `app/sw.ts` instead:
```typescript
/// <reference no-default-lib="true" />
/// <reference lib="esnext" />
/// <reference lib="webworker" />
```
Do NOT add `"webworker"` to the main `tsconfig.json` lib. Add `public/sw.js` (if generated) to `tsconfig.json` `exclude` array. The `@serwist/next/typings` addition to `types` is for webpack mode only — not needed for Turbopack mode.

## Code Examples

### Offline Connectivity Toast

```typescript
// components/pwa/offline-toast.tsx
"use client";
import { useEffect, useRef } from "react";
import { toast } from "sonner";
import { useOnlineStatus } from "@/hooks/use-online-status";

export function OfflineToast() {
  const isOnline = useOnlineStatus();
  const hasMounted = useRef(false);

  useEffect(() => {
    if (!hasMounted.current) {
      hasMounted.current = true;
      return; // don't fire on initial mount
    }
    if (isOnline) {
      toast.success("Back online");
    } else {
      toast.warning("You're offline", {
        description: "Cached content is still available.",
        duration: Infinity, // persist until back online
        id: "offline-toast",
      });
    }
  }, [isOnline]);

  return null;
}
```

## Discretion Recommendations

Based on research, here are recommendations for the discretion items:

**Install Prompt UI:** Show a subtle bottom banner (not modal) only on Chrome/Android using `beforeinstallprompt`. Dismiss on install or explicit close. Skip iOS (no API available — don't show broken UI). Timing: delay 30 seconds after first meaningful interaction to avoid annoying users on first visit.

**Offline Mutation Strategy:** Read-only when offline. Attempting to queue-and-sync mutations requires IndexedDB + background sync, adding significant complexity. For a watchlist/library app, read-only is acceptable UX — show disabled state on write buttons with "Available when online" tooltip.

**AI Mood Section When Offline:** Show disabled state, not hidden. Keep the input visible but disable it with an inline "AI recommendations require internet" message. Hiding it entirely is disorienting for installed PWA users who know the feature exists.

**API Response Caching:** Use `NetworkFirst` for `/api/movies/*` (fresh content matters for rankings) and `/api/ai/*` (streaming, never cache). Use `StaleWhileRevalidate` for the TMDB proxy routes that return paginated category lists (trending, popular) — acceptable to serve stale for offline browsing. Never cache auth (`/api/auth/*`) or AI streaming routes.

**Status Bar / Theme Color:** Use dark background `#09090b` for `theme_color` in manifest — matches the standalone window chrome. Crimson would look odd as the browser chrome/toolbar color. Add `apple-mobile-web-app-status-bar-style: black-translucent` for iOS.

**Cache Size Limit:** Use `maxEntries: 500` for TMDB images (~100MB at 200KB average poster). This fits the "50-100MB range" decision and enforces LRU eviction automatically.

**Splash Screen Animation:** Use a CSS fade-in on the centered Moodflix wordmark + a subtle pulsing crimson glow. Keep it under 300ms — the splash is transient, not a loading screen.

## Open Questions

1. **Sentry + Serwist Turbopack coexistence in prod build**
   - What we know: Sentry uses webpack plugin via `withSentryConfig`. `@serwist/turbopack` uses route handlers (no webpack involvement). Both should coexist cleanly.
   - What's unclear: Whether `@serwist/turbopack` needs any `next.config.ts` changes that could conflict with `withSentryConfig`.
   - Recommendation: Test `npm run build` after installing `@serwist/turbopack` and wrapping config. If conflicts arise, the fallback is to use `@serwist/next` with `--webpack` and remove Sentry's webpack config from build (Sentry's source maps are not critical for MVP).

2. **Icon Generation Tooling**
   - What we know: `sharp` is already installed (via Next.js `@img/sharp-darwin-arm64`). Existing PNGs are 192x192 and 512x512.
   - What's unclear: Whether to generate icons via a Node.js script using sharp, a Figma/design tool export, or an online generator like maskable.app.
   - Recommendation: The "cinematic crimson & dark" icon is a design task, not purely code.
