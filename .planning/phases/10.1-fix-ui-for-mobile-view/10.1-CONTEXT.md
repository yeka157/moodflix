# Phase 10.1: Fix UI for Mobile View - Context

**Gathered:** 2026-03-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix 4 specific mobile UI bugs: cropped hero text on landing page, duplicate carousel rows on landing page mobile fallback, rigid animations across the app, and missing horizontal padding on content pages. No redesign — fix what's broken on the current dark theme with crimson accent.

</domain>

<decisions>
## Implementation Decisions

### Landing page hero text
- Hero text is cropped on mobile viewports — must be fully visible
- Claude decides the best approach (scale down, reflow, or responsive typography) — just make it look good on mobile

### Landing page poster/carousel
- The 3D poster component is currently hidden on mobile, falling back to an old carousel that shows 2 duplicate rows of the same movies
- User wants the 3D poster on mobile if it works well — if not, Claude decides between a single-row carousel or 2 rows of different (non-duplicate) movies
- Priority: no duplicate content, whatever approach looks best on small screens

### Animation quality
- All animations across the app feel rigid and not smooth on mobile — this is a global issue, not page-specific
- Audit and fix motion timing, easing curves, and duration across all Framer Motion animations
- Reduce animation complexity on mobile if needed for performance (respect `prefers-reduced-motion`)
- Goal: fluid, natural-feeling transitions — not stiff or mechanical

### Content padding
- Consistent horizontal padding on ALL content across all pages on mobile
- No edge-to-edge content — everything gets uniform padding
- Applies to: home page, discover page, watchlist, series, library, settings, detail pages

### Claude's Discretion
- Exact padding values (16px, 20px, etc.) — whatever looks balanced
- Specific animation easing curves and durations
- Whether 3D poster works on mobile or needs alternative
- Hero text responsive strategy (font scaling vs line wrapping)
- Any additional mobile layout fixes discovered during implementation

</decisions>
