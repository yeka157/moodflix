## Summary

This phase fixes four concrete mobile UI regressions across the Moodflix app. The issues are all locatable in existing source files тАФ no new infrastructure is needed, only targeted Tailwind class changes, Framer Motion prop adjustments, and one structural fix to the landing page showcase fallback.

The most impactful fix is the hero text crop: `hero-section.tsx` uses `text-5xl lg:text-6xl xl:text-7xl` тАФ the `text-5xl` (48px) base size is too large for small viewports with a full side-by-side layout (`flex-col lg:flex-row`). On a 375px viewport with `px-4` padding, 48px text at 1.08 line-height clips against the top of the sticky viewport when `pt-24` pushes it down. The fix is responsive type scaling starting from a smaller mobile base (`text-3xl sm:text-4xl lg:text-6xl xl:text-7xl`) and ensuring the hero sticky height has room.

The duplicate carousel is a straightforward bug in `movie-showcase.tsx`: `MarqueeFallback` uses `const row2 = [...posters, ...posters]` where both `row1` and `row2` duplicate the same `posters` array. Both rows scroll the same content. The fix is to split `posters` into two halves and fill each row from alternating halves, or тАФ since the user's preferred solution is 3D on mobile if it works тАФ evaluate whether WebGL/Three.js runs acceptably on mobile.

The animation rigidity is caused by spring configs with high stiffness (300тАУ500) and low damping (15тАУ20) which produce snappy, mechanical motion. On mobile this feels harsh because touch interactions have higher latency. The fix is softer springs (`stiffness: 200, damping: 25`) and slight easing-out curve adjustments on enter/exit variants.

The padding issue is partially already present: most pages use `px-4 sm:px-6 lg:px-8` at the page level. However the `HeroBanner` component breaks this with `-mx-4 -mt-8` (intentional negative margin to bleed the banner), and movie/TV detail pages use `max-w-5xl mx-auto px-4 md:px-6` тАФ which has no small-viewport padding at the `sm:` breakpoint, potentially causing some tightness. The layout wrapper `main` has `pb-16 md:pb-0` but no horizontal padding тАФ pages must self-apply it.

**Primary recommendation:** Fix four isolated issues in six files тАФ no architectural changes needed. The MarqueeFallback duplicate-row bug is the simplest fix; hero text is a type scale change; animation improvements are prop-by-prop Spring config updates; padding consistency requires auditing each page's outermost container class.

### Pattern 1: Responsive Type Scale (hero fix)

**What:** Replace flat `text-5xl` mobile base with graduated scale
**When to use:** Any headline visible on mobile that uses large type

Current (broken):
```tsx
// hero-section.tsx line 385
<motion.h1
  className="text-5xl lg:text-6xl xl:text-7xl font-bold tracking-tight mb-6 leading-[1.08]"
```

Fix:
```tsx
<motion.h1
  className="text-3xl sm:text-4xl lg:text-6xl xl:text-7xl font-bold tracking-tight mb-6 leading-[1.08]"
```

Also: the hero content area uses `flex flex-col lg:flex-row` тАФ on mobile the MockupFrame renders below the text but still occupies significant viewport height. The sticky viewport is exactly `h-screen`. With `pt-24` (96px) from top and the stacked layout, the text block can push the CTA buttons off-screen. Options:
1. On mobile, hide the MockupFrame entirely (`hidden lg:flex`), giving the text full width and room to breathe
2. Scale down `MockupFrame` on mobile with `w-full max-w-xs mx-auto` and reduce its aspect ratio

Option 1 (hide mockup on mobile) is cleaner тАФ the mockup is decorative, not functional.

### Pattern 2: MarqueeFallback Duplicate-Row Fix

**What:** Both rows in `MarqueeFallback` duplicate the same `posters` array
**Root cause:** `const row2 = [...posters, ...posters]` тАФ same content as `row1`

```tsx
// Current (broken) тАФ movie-showcase.tsx line 75-76
const row1 = [...posters, ...posters];
const row2 = [...posters, ...posters];
```

Fix A тАФ Two halves of distinct movies:
```tsx
const half = Math.ceil(posters.length / 2);
const firstHalf = posters.slice(0, half);
const secondHalf = posters.slice(half);
// Each row still needs doubling for seamless marquee loop
const row1 = [...firstHalf, ...firstHalf];
const row2 = [...secondHalf, ...secondHalf];
```

Fix B тАФ 3D on mobile (preferred by user if viable):
Remove the `isMobile` condition in `MovieShowcase`. Three.js/WebGL works on modern mobile browsers (iOS Safari 15+, Android Chrome). The `dpr={[1, 1.5]}` cap in `MovieShowcase3D` already limits GPU load. However, the 3D experience requires users to scroll through `h-[500vh]` тАФ on mobile this consumes the entire landing page scroll. This is a UX concern: mobile users would have to scroll 5 viewport-heights through the 3D section before reaching Features, AI Preview, and CTA.

**Recommendation:** Use Fix A (split into two distinct rows). Keep the `isMobile` check so mobile gets the marquee, which is lighter and shows the content without requiring 5-viewport scroll. Two distinct halves eliminates the duplicate content complaint. If posters.length < 4, fall back to a single marquee row.

### Pattern 3: Framer Motion Spring Tuning

**What:** Global animation feel improvement тАФ softer springs, smoother easing
**Files affected:** movie-card.tsx, movie-grid.tsx, hero-section.tsx, hero-banner.tsx, movie-detail-page.tsx, watchlist-content.tsx

**Current spring configs (rigid):**
- `movie-card.tsx` whileHover: `{ type: "spring", stiffness: 300, damping: 20 }` тЖТ hover scale
- `movie-card.tsx` icon appear: `{ type: "spring", stiffness: 500, damping: 15 }` тЖТ very snappy
- `movie-grid.tsx` whileInView: `{ duration: 0.4 }` (linear-ish default ease)
- `hero-banner.tsx` items: `{ duration: 0.45, ease: "easeOut" }` тЖТ acceptable
- `movie-detail-page.tsx` backdrop: `{ duration: 0.6, ease: "easeOut" }` тЖТ acceptable

**Recommended replacements:**

```tsx
// movie-card.tsx тАФ whileHover scale (smoother, less mechanical)
transition={{ type: "spring", stiffness: 200, damping: 25 }}

// movie-card.tsx тАФ icon appear animation (less snappy)
transition={{ type: "spring", stiffness: 300, damping: 22 }}

// movie-grid.tsx тАФ whileInView entry (softer entrance)
transition={{ duration: 0.5, ease: [0.25, 0.1, 0.25, 1] }}  // cubic-bezier ease-in-out

// hero-section.tsx scroll-pinned exits тАФ y translations feel linear
// Currently: y: [0, 0, -60] тАФ consider cubic easing for the exit
// These use useTransform тАФ no direct ease prop, but they're scroll-driven so
// the "rigidity" is from the linear scroll тЖТ linear motion mapping.
// No fix needed here specifically тАФ this is intentional scroll pinning.
```

**Mobile-specific:** Add `whileHover` guard for touch devices. Hover animations don't trigger on touch, but `whileTap` does. The current `tapAnimation = prefersReducedMotion ? {} : { scale: 0.85 }` is good. Keep it.

**Key insight:** Framer Motion spring `stiffness: 500, damping: 15` produces 3-4 rapid oscillations before settling тАФ this reads as "bouncy and mechanical." `stiffness: 250-300, damping: 22-28` produces one clean overshoot or direct settle, which feels natural.

### Anti-Patterns to Avoid
- **Adding `overflow-hidden` to fix text crop** тАФ hides text rather than fixing layout
- **Using `min-h-0` hacks instead of fixing type scale** тАФ creates brittle constraints
- **Removing 3D showcase entirely** тАФ only remove the mobile condition; desktop keeps 3D
- **Using `!important` with Tailwind** тАФ never needed with proper breakpoint specificity
- **Animating `width` or `height` instead of `scale`** тАФ triggers layout, janky on mobile
- **Jumping to `whileHover={{ scale: 0 }}`** to "disable" hover on mobile тАФ wrong, use media queries or pointer checks

## Common Pitfalls

### Pitfall 1: Hero Sticky Layout Height
**What goes wrong:** The sticky viewport is `h-screen`. On mobile with `flex-col` stacking, the text + mockup frame exceed viewport height. The overflow is hidden by `overflow-hidden` on the sticky container, clipping the bottom content (including CTA buttons).
**Why it happens:** `h-screen` + `overflow-hidden` + stacked flex children + `pt-24` (96px top padding) leaves ~60тАУ70% of the viewport for two stacked blocks (text + mockup), but the mockup's `aspect-video` ratio + full width takes about 56vw height on mobile тАФ too tall.
**How to avoid:** Hide the MockupFrame on mobile (`hidden lg:block`). The text block alone at 3xlтАУ4xl size fits easily within the remaining ~75vh.
**Warning signs:** `pt-24` (96px) + text block + mockup > 100vh тЖТ anything at the bottom clips.

### Pitfall 2: MarqueeFallback Row Content Duplication
**What goes wrong:** Both rows show identical poster sequences. Users immediately see the repetition.
**Why it happens:** Both `row1` and `row2` reference the same `posters` array with the same doubling pattern. The second row was likely intended to scroll in the opposite direction (marquee-right) with different content for visual variety.
**How to avoid:** Split the `posters` array into non-overlapping halves. Verify `posters.length >= 4` before applying; with fewer than 4, single-row is fine.
**Warning signs:** Row 2 starts with the same poster as Row 1.

### Pitfall 3: Spring Overshoot on Small Tap Targets
**What goes wrong:** `whileTap={{ scale: 0.85 }}` on a small button (32├Ч32px) can feel harsh and unresponsive on mobile if combined with a high-stiffness spring.
**Why it happens:** The default spring for whileTap is snappy тАФ combined with `stiffness: 500`, the scale animation completes in < 100ms, shorter than a typical human tap duration, so it looks like it didn't respond.
**How to avoid:** Use `whileTap={{ scale: 0.92 }}` (gentler compression) with `transition={{ duration: 0.1 }}` for icon buttons. The `0.85` is more appropriate for larger CTA buttons.
**Warning signs:** Buttons that feel "laggy" or "invisible" on mobile despite responding correctly.

### Pitfall 4: Detail Page Poster Width on Small Screens
**What goes wrong:** The movie/TV detail page poster is `hidden sm:block` тАФ it disappears on screens < 640px. This is intentional. But the title/info area gets full width without `min-w-0`, causing overflow if a very long title pushes beyond the container.
**Why it happens:** `min-w-0` is required on flex children that hold text that should truncate/wrap.
**How to avoid:** The current code uses `<div className="min-w-0 pb-1">` тАФ already correct. No fix needed here, but confirm it stays on any edits to that section.

### Pitfall 5: Tailwind v4 Negative Margin Interactions
**What goes wrong:** `HeroBanner` uses `-mx-4 -mt-8` to bleed edge-to-edge. If the parent has responsive padding `px-4 sm:px-6`, the negative margin only cancels `px-4` on mobile. On `sm:` breakpoints, `-mx-4` undercorrects and leaves a 8px gap (since parent has `px-6` = 24px but `-mx-4` = -16px).
**Why it happens:** Negative margin is hardcoded to `4` units but parent padding varies by breakpoint.
**Current state:** `HeroBanner` is inside a div with `px-4 sm:px-6 lg:px-8` тАФ so the hero will have visible left/right margins at sm+ breakpoints. The current code uses `-mx-4` only, so this gap already exists. However this is a full-bleed hero with a backdrop image тАФ a slight contained look at sm+ may actually be acceptable (or was not noticed).
**How to avoid:** Use `-mx-4 sm:-mx-6 lg:-mx-8` to match parent padding at each breakpoint, OR move the `HeroBanner` outside the padded wrapper. The cleaner fix is to move `HeroBanner` outside the padded `div` in `home/page.tsx`:

```tsx
// home/page.tsx current:
<div className="space-y-10">
  {featuredMovie && <HeroBanner movie={featuredMovie} />}  {/* inside padding */}
  <div className="px-4 sm:px-6 lg:px-8 space-y-10">
    ...
  </div>
</div>

// Fix: HeroBanner is already OUTSIDE the padded div тАФ and uses -mx-4 to escape
// its implicit parent container. The parent div has no padding, so -mx-4 isn't
// needed for the bleed. Actually, the -mx-4 is compensating for the space-y-10
// container margin which doesn't exist... Let's verify the nesting:
```

**Clarification:** In `home/page.tsx`, `HeroBanner` is inside the outer `<div className="space-y-10">` which has NO horizontal padding. Then the padded `<div className="px-4 sm:px-6 lg:px-8 space-y-10">` is a sibling. The `-mx-4` in `HeroBanner` is compensating for something that isn't there (the outer div has no px). This means `-mx-4 -mt-8` effectively gives the hero a negative left/right bleed on mobile (going 16px beyond both edges) тАФ which may have been intended for full-bleed. This is already working correctly тАФ no fix needed.
