# Phase 12.2: Content Discovery & AI Shazam - Research

**Researched:** 2026-03-07
**Domain:** TMDB API caching, Coming Soon UI, AI tool-calling for media identification
**Confidence:** HIGH

## Summary

This phase adds three features: Coming Soon rows using TMDB `/movie/upcoming` and `/tv/on_the_air` endpoints, a Supabase-backed TMDB caching layer with tiered TTLs, and an AI "Shazam" feature that identifies movies/TV shows from natural language descriptions. All three integrate into existing patterns -- MovieRow for Coming Soon, Drizzle schema for caching, and the existing AI chat route for Shazam.

The caching layer is the most architecturally significant change. It introduces two new Drizzle tables (`tmdb_cache` for category lists, `tmdb_media` for individual titles) and wraps all existing `lib/tmdb.ts` functions with a read-through cache. The AI Shazam extends the existing `suggest_genres` tool pattern with a new `identify_media` tool that returns TMDB IDs, rendered as inline cards in the mood chat.

**Primary recommendation:** Implement caching layer first (schema + wrapper functions), then Coming Soon rows (which benefit from caching), then AI Shazam (independent feature).

### Pattern 1: Lazy Cache Read-Through
**What:** Every TMDB data function gets a cache-aware wrapper that checks DB first, falls back to TMDB API, then stores result.
**When to use:** All category list fetches and detail page fetches (NOT search).
**Example:**
```typescript
// lib/tmdb-cache.ts
export async function getCachedTrending(page = 1): Promise<MovieListResponse> {
  const cacheKey = `trending_movie_week_${page}`;

  // 1. Check cache
  const cached = await db.select()
    .from(tmdbCache)
    .where(eq(tmdbCache.key, cacheKey))
    .limit(1);

  if (cached.length > 0 && !isStale(cached[0].updatedAt, 24 * 60 * 60 * 1000)) {
    return cached[0].data as MovieListResponse;
  }

  // 2. Fetch from TMDB
  const fresh = await getTrendingMovies(page);

  // 3. Upsert to cache (fire-and-forget)
  db.insert(tmdbCache)
    .values({ key: cacheKey, category: "trending", mediaType: "movie", data: fresh, updatedAt: new Date() })
    .onConflictDoUpdate({ target: [tmdbCache.key], set: { data: fresh, updatedAt: new Date() } })
    .catch(() => {});

  return fresh;
}
```

### Pattern 2: AI Tool for Media Identification
**What:** New `identify_media` tool alongside existing `suggest_genres`. Gemini identifies the movie/TV show from description, returns title and TMDB ID.
**When to use:** When user describes a specific scene, plot, or character rather than a mood.
**Example:**
```typescript
identify_media: tool({
  description: "Identify a specific movie or TV show from a user's description of its plot, scenes, or characters.",
  inputSchema: z.object({
    title: z.string().describe("The identified movie or TV show title"),
    tmdbId: z.number().describe("The TMDB ID of the identified title"),
    mediaType: z.enum(["movie", "tv"]).describe("Whether it's a movie or TV show"),
    year: z.string().optional().describe("Release year if known"),
    confidence: z.enum(["high", "medium", "low"]).describe("How confident the identification is"),
  }),
  execute: async (params) => {
    // Verify the TMDB ID exists by searching
    const results = await searchMovies(params.title, 1);
    const match = results.results.find(m => m.id === params.tmdbId);
    return {
      ...params,
      verified: !!match,
      posterPath: match?.poster_path ?? null,
      overview: match?.overview ?? null,
    };
  },
}),
```

### Anti-Patterns to Avoid
- **Caching search results:** User decided search stays live. Do not cache `/search/*` endpoints.
- **Eager cache population:** Use lazy/on-demand caching only. No background jobs or cron.
- **Blocking on cache writes:** All DB cache writes must be fire-and-forget to avoid adding latency.
- **Gemini hallucinating TMDB IDs:** The AI will guess TMDB IDs incorrectly. The `identify_media` execute function MUST verify via TMDB search. If the ID doesn't match, search by title and return the best match.
- **Separate cache invalidation service:** TTL is checked at read time via timestamp comparison. No scheduled cleanup needed.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| TMDB ID verification | Manual ID lookup | TMDB `/search/multi` endpoint | Returns both movies and TV in one call, includes media_type field |
| Cache TTL management | Cron job or scheduled function | Timestamp comparison at query time | Simpler, no infrastructure needed, stale-while-revalidate pattern |
| Date formatting for badges | Manual date parsing | `Intl.DateTimeFormat` with `{ month: 'short', day: 'numeric' }` | Locale-aware, handles edge cases |
| Tool result rendering | Custom message parser | AI SDK v5 `part.type.startsWith("tool-")` pattern | Already established in mood-section.tsx |

## Common Pitfalls

### Pitfall 1: Gemini Hallucinating TMDB IDs
**What goes wrong:** Gemini will confidently return fake TMDB IDs that don't correspond to any movie.
**Why it happens:** LLMs don't have reliable knowledge of database-specific identifiers.
**How to avoid:** The `identify_media` tool's `execute` function must search TMDB by title (not trust the ID). Use `/search/multi` to find the actual TMDB entry and return verified data.
**Warning signs:** Movie cards showing "N/A" or broken poster images.

### Pitfall 2: TMDB Upcoming Returns Past Dates
**What goes wrong:** `/movie/upcoming` returns movies with release dates in the recent past (within ~2 weeks), not just future releases.
**Why it happens:** TMDB defines "upcoming" as a rolling window, not strictly future-only.
**How to avoid:** Filter results client-side: only show movies where `release_date > today`. Display date badge for all, but only label "Coming Soon" for future dates.
**Warning signs:** "Coming Soon" row showing movies already in theaters.

### Pitfall 3: Cache Table Bloat
**What goes wrong:** `tmdb_cache` accumulates stale category list entries that are never cleaned up.
**Why it happens:** Lazy cache only checks TTL on read, never deletes old entries.
**How to avoid:** Use `ON CONFLICT DO UPDATE` (upsert) pattern so each cache key has at most one row. Category keys are deterministic (e.g., `trending_movie_week_1`), so the table stays bounded.
**Warning signs:** Slow cache queries, growing table size.

### Pitfall 4: Race Condition on Cache Write
**What goes wrong:** Two concurrent requests both see a cache miss and both write to the DB.
**Why it happens:** No locking between read and write in the lazy cache pattern.
**How to avoid:** Use `ON CONFLICT DO UPDATE` for all cache upserts. The second write simply overwrites with equally fresh data. This is acceptable because TMDB responses are idempotent.
**Warning signs:** Unique constraint violations in logs (but upsert prevents this).

### Pitfall 5: AI Intent Detection Ambiguity
**What goes wrong:** Gemini calls `identify_media` for mood requests or `suggest_genres` for identification requests.
**Why it happens:** Some queries are ambiguous (e.g., "something like The Matrix" could be mood or identification).
**How to avoid:** System prompt should instruct: if the user names a specific title or describes a specific plot/scene, use `identify_media`. If they describe a mood, feeling, or vibe, use `suggest_genres`. For "something like X", use `suggest_genres` with genres matching X.
**Warning signs:** Users getting identification cards when they wanted genre suggestions.

### Pitfall 6: normalizeTVShow Drops release_date for Coming Soon Badge
**What goes wrong:** `normalizeTVShow` maps `first_air_date` to `release_date` on the Movie type, but the Coming Soon badge formatter needs to know the original field.
**Why it happens:** The normalized Movie type uses `release_date` for both movies and TV.
**How to avoid:** This is fine -- `release_date` field on the normalized Movie works for the badge. Just ensure `first_air_date` is present in the TMDB response (it always is for `/tv/on_the_air`).

export const tmdbCache = pgTable("tmdb_cache", {
  id: uuid("id").primaryKey().defaultRandom(),

### TTL Check Helper
```typescript
// lib/tmdb-cache.ts
const TTL = {
  CATEGORY: 24 * 60 * 60 * 1000,    // 24 hours
  RATINGS: 7 * 24 * 60 * 60 * 1000, // 7 days
  DETAILS: Infinity,                  // Permanent (never stale)
} as const;

function isStale(updatedAt: Date | null, ttlMs: number): boolean {
  if (!updatedAt) return true;
  if (ttlMs === Infinity) return false;
  return Date.now() - updatedAt.getTime() > ttlMs;
}
```

// lib/utils.ts addition
export function formatReleaseDateBadge(dateStr: string): string | null {
  if (!dateStr) return null;
  const date = new Date(dateStr + "T00:00:00"); // Avoid timezone shift
  const now = new Date();
  if (date <= now) return null; // Only show for future dates
  return new Intl.DateTimeFormat("en-US", { month: "short", day: "numeric" }).format(date);
}

## Open Questions

1. **TMDB ID verification strategy**
   - What we know: Gemini will hallucinate TMDB IDs. We need to verify via search.
   - What's unclear: Should `identify_media.execute` use `/search/multi` (one call, both types) or separate `/search/movie` and `/search/tv` based on `mediaType`?
   - Recommendation: Use `/search/multi` -- single API call, returns `media_type` field, handles cases where Gemini guesses wrong type.

2. **Cache migration strategy**
   - What we know: All existing TMDB calls in page components use direct functions from `lib/tmdb.ts`.
   - What's unclear: Should we swap all callers in one plan or incrementally?
   - Recommendation: Create `lib/tmdb-cache.ts` with identical function signatures (e.g., `getCachedTrending`). Swap callers in page components incrementally. This avoids a big-bang migration.

3. **`tmdb_media` detail granularity**
   - What we know: User wants watchlist cards to never need separate TMDB calls for basic info.
   - What's unclear: How much detail data to store (just basic fields, or full credits/providers)?
   - Recommendation: Store basic fields (title, poster, year, rating, genres) as individual columns for direct queries. Store full details response as JSONB `details_data` column for detail pages. Two-level: basic is always populated on first encounter, details populated when detail page is viewed.
