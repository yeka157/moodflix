Add Coming Soon rows to /home, /discover, and /series pages. Implement a TMDB lazy caching layer in Supabase for category lists and individual movie/TV details. Add AI "Shazam for movies" feature that identifies movies/TV shows from user descriptions, integrated into the existing mood chat.

<decisions>
## Implementation Decisions

### Coming Soon Rows
- Add a Coming Soon row to all three pages: /home, /discover (Movies), and /series — each shows its own media type (movies vs TV)
- Position: after trending row, before popular row on each page
- Show a release date badge on each card (e.g., "Mar 28") — distinguishes Coming Soon cards from regular browse cards
- Reuse existing bookmark overlay — users can add upcoming titles to their watchlist directly from the row
- Use TMDB `/movie/upcoming` and `/tv/on_the_air` endpoints as data sources

### TMDB Caching Layer
- Goal: reduce direct TMDB API calls by caching responses in Supabase
- Two-tier caching strategy:
  - **Category lists** (trending, popular, upcoming, top_rated): 24-hour TTL — re-fetch daily when stale
  - **Movie/TV details** (title, overview, cast, poster path, etc.): Store permanently — this data rarely changes
  - **Ratings** (vote_average, vote_count): Weekly TTL — ratings shift slowly
- Lazy cache pattern: first request fetches from TMDB and saves to DB. Subsequent requests serve from DB. When TTL expires, next request triggers a background refresh from TMDB
- Cache powers everything: watchlist cards, detail pages, and browse rows all read from cached data instead of hitting TMDB directly
- Search stays live — no caching for search results (too many unique queries)
- Schema: `tmdb_cache` table for category list responses, `tmdb_media` table for individual movie/TV details

### AI Shazam (Movie Identification)
- Integrated into the existing mood chat on /home — no separate page or section
- AI automatically detects intent: mood descriptions trigger genre suggestions (existing flow), specific movie/TV descriptions trigger identification (new flow)
- System prompt instructs Gemini to distinguish between mood requests ("I feel sad") and identification requests ("the movie about a ring")
- New tool call (e.g., `identify_media`) returns identified movie/TV with TMDB ID
- Result displayed as an inline card in the chat: poster, title, year, and quick actions (view details, add to watchlist)
- When AI can't identify the exact title, fall back to suggesting similar movies based on the description (rather than asking clarifying questions)
- Identification results logged to `ai_conversations` using same fire-and-forget pattern

### Claude's Discretion
- Database schema design for `tmdb_cache` and `tmdb_media` tables (columns, indexes)
- How to handle TTL checking (timestamp comparison in query vs scheduled cleanup)
- Exact system prompt wording for intent detection (mood vs identification)
- Inline card component design for Shazam results (layout, animations)
- How to surface similar movies when identification fails (genre-based fallback vs description-based search)
- Migration strategy for transitioning existing TMDB-direct calls to cached reads

</decisions>

<specifics>
## Specific Ideas

- Coming Soon row should feel like the other browse rows (same horizontal scroll, same arrow buttons) with the addition of a date badge — consistency is key
- The caching layer should be transparent to the UI — components don't need to know whether data comes from TMDB or cache
- AI Shazam should feel like a natural extension of the mood chat, not a mode switch — "the one where the astronaut grows potatoes on Mars" should just work
- Cache should store enough detail that watchlist cards never need a separate TMDB call for basic info (poster, title, year)
- User specifically mentioned wanting movie details stored in own DB, not just category lists

</specifics>
