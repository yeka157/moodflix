## Ticket-by-Ticket Analysis

### BACKLOG-20: Gemini Quota Exceeded

**Root cause:** `gemini-2.0-flash` is being retired March 3, 2026 and has essentially zero free-tier quota remaining after Google's December 2025 rate limit reduction.

**Current state:** `lib/ai.ts` uses `google("gemini-2.0-flash")`. When quota is exceeded, the error propagates as a generic 500 from the `catch` block in the route handler — the user sees "Failed to generate recommendations. Please try again." This is not descriptive enough.

**Fix:** Switch to `gemini-2.5-flash-lite` (1,000 RPD on free tier vs. effectively 0 for `gemini-2.0-flash`). Also add specific error detection for quota/rate-limit errors (HTTP 429/503) and return a user-facing 429 with a clear message.

**Model IDs (HIGH confidence — ai-sdk.dev docs):**
- `gemini-2.5-flash-lite` — 15 RPM, 1,000 RPD on free tier
- `gemini-2.5-flash` — 10 RPM, 250 RPD on free tier (fallback option)
- `gemini-2.5-flash-lite` is the correct primary choice for high-volume free use

**`lib/ai.ts` change:**
```typescript
// Source: https://ai-sdk.dev/providers/ai-sdk-providers/google-generative-ai
export const geminiModel = google("gemini-2.5-flash-lite");
```

**Error detection in route handler:**
```typescript
// In the catch(err) block in app/api/ai/recommend/route.ts
import { AISDKError } from "ai";

// Check if it's an API call error with 429/503 status
function isQuotaError(err: unknown): boolean {
  if (err instanceof Error) {
    const msg = err.message.toLowerCase();
    if (msg.includes("quota") || msg.includes("rate limit") || msg.includes("429") || msg.includes("resource exhausted")) {
      return true;
    }
  }
  // AI SDK wraps provider errors — check cause chain
  if (err && typeof err === "object" && "cause" in err) {
    return isQuotaError((err as { cause: unknown }).cause);
  }
  return false;
}
```

**Route handler error response for quota:**
```typescript
if (isQuotaError(err)) {
  return Response.json(
    { error: "AI recommendations are temporarily unavailable. Please try again later." },
    { status: 503 }
  );
}
```

### BACKLOG-27: AI Retry Cap / Budget Burn

**Root cause:** `stopWhen: stepCountIs(3)` allows up to 3 AI steps, but if the model keeps calling tools or failing, we're burning tokens. The app already has `stopWhen: stepCountIs(3)` which limits steps — but the issue is retries at the *HTTP level* (the AI SDK retries failed requests by default).

**Current state:** AI SDK v6 `streamText` by default retries transient errors. When Gemini returns 429 or 503, the SDK may retry automatically before the error surfaces. This burns quota silently.

**Fix:** Set `maxRetries: 0` on the `streamText` call. The rate-limiter (10/day per user) already prevents budget burn from the user side. The concern is the AI SDK retrying the *provider* — we want to fail fast and show a friendly error rather than burn quota on retries.

```typescript
// Source: https://ai-sdk.dev/docs/ai-sdk-core/settings
const result = streamText({
  model: geminiModel,
  system: systemPrompt,
  messages: modelMessages,
  tools: { ... },
  stopWhen: stepCountIs(3),
  maxOutputTokens: 1000,
  maxRetries: 0,  // ADD THIS — fail fast, don't burn quota on retries
});
```

Also add `onError` callback to log the actual provider error:
```typescript
const result = streamText({
  // ...
  maxRetries: 0,
  onError: (event) => {
    console.error("AI stream error:", event.error);
  },
});
```

### BACKLOG-28: K-drama / C-drama / Regular TV Separation

**Current state analysis:** The `/series` page (Phase 7 revamp) shows `SeriesGridContent` — a filter-first grid with Genre, Sort, Year selects. The old `SeriesContent` component (with separate MovieRow rows for Trending, Airing Today, Korean, Chinese, Top Rated) still exists but is **not used** in the current `/series/page.tsx`.

The Phase 7 revamp replaced the curated-rows layout with the filter-driven grid. The current series page is a uniform grid where all shows are mixed. BACKLOG-28 is valid: K-dramas, C-dramas, and regular TV shows appear together with no visual separation.

**Two viable approaches:**

**Option A: Restore curated rows above the filter grid**
- Keep `SeriesGridContent` for filtered browsing
- Add back `SeriesContent` rows (Trending, Korean Drama, Chinese Drama, Top Rated) as the "Browse" section above the grid
- This adds back clear category rows without removing the filter grid

**Option B: Add a "Type" filter to the existing grid**
- Add a new filter: "All Series | Korean Drama | Chinese Drama | Regular TV"
- Korean Drama filter: `with_origin_country=KR&with_original_language=ko`
- Chinese Drama filter: `with_origin_country=CN` (without language — spans Mandarin/Cantonese)
- Regular TV: exclude Korean/Chinese by `without_keywords` or just "All" minus drama filters

**Recommendation:** Option A. It gives users the clearest separation and matches the user's expectation of "seeing Korean dramas as a distinct row." The curated rows already exist in `SeriesContent`. The page just needs to bring them back above the filter grid. This is lower risk than adding complex filter exclusion logic.

**Implementation:** Replace `SeriesPage` to render both `SeriesContent` (SSR rows) and `SeriesGridContent` (interactive filter grid), similar to the old `app/(app)/series/page.tsx` structure from Phase 5.

### BACKLOG-30: Infinite Scroll Incomplete Last Row

**Root cause:** `MovieGrid` uses `grid-cols-2 sm:grid-cols-3 md:grid-cols-4 lg:grid-cols-5 xl:grid-cols-6`. At 1280px+ you get 6 columns. If TMDB returns 20 results per page, the last row has 2 orphan items (20 mod 6 = 2). Infinite scroll triggers at 400px before sentinel — so partial rows are visible before next page loads.

**The real issue:** When infinite scroll has loaded all available results (no next page), the final partial row is permanent. The user sees an uneven grid.

**Fix approach — pad to column multiple:**

In `SeriesGridContent` (and potentially `discover-content.tsx`), after deduplication, pad the `shows` array to the nearest column-count multiple with placeholder/null items, and render invisible placeholder cards for the padding items.

**Column counts per breakpoint:**
- default: 2
- sm (640px): 3
- md (768px): 4
- lg (1024px): 5
- xl (1280px+): 6

**Problem:** We don't know the user's current breakpoint in a static way. The cleanest CSS-only solution is to use a `visibility: hidden` padding approach — always pad to 6 (the max column count) since excess padding cards in narrower viewports don't create a new row they just fill within existing rows.

**Practical approach:** Only pad when `!hasNextPage` (infinite scroll is done). Pad to the next multiple of 6.

```typescript
// In SeriesGridContent, after shows = useMemo(...)
const paddedShows = useMemo(() => {
  if (discoverQuery.hasNextPage) return shows; // Don't pad while more pages exist
  const remainder = shows.length % 6;
  if (remainder === 0) return shows;
  const padCount = 6 - remainder;
  return [...shows, ...Array(padCount).fill(null)];
}, [shows, discoverQuery.hasNextPage]);
```

In `MovieGrid`, handle `null` items:
```typescript
{(movie === null) ? (
  <div key={`pad-${i}`} className="invisible" aria-hidden="true" />
) : (
  <motion.div key={movie.id}>
    <MovieCard ... />
  </motion.div>
)}
```

**Simpler alternative:** Use CSS `grid-auto-flow: dense` — but this reorders items. Not suitable.

**Best alternative: CSS only approach** — use `content-visibility` hidden pseudo-items via `:after` and `:before` tricks. These are complex and less readable than the JS pad approach.

**Recommendation:** JS padding with `null` sentinel values, rendered as invisible `div` elements. This is readable, maintainable, and works with the existing grid structure.

## Common Pitfalls

### Pitfall 1: gemini-2.0-flash Retirement
**What goes wrong:** Model retired March 3, 2026 — requests fail with quota/auth errors.
**Why it happens:** Google's free tier limits were decimated in Dec 2025; 2.0-flash is end-of-life.
**How to avoid:** Switch to `gemini-2.5-flash-lite` (1,000 RPD free tier).
**Warning signs:** Console shows "quota exceeded" with 0 usage in Google AI Studio.

### Pitfall 2: AI SDK Default Retries Burning Quota
**What goes wrong:** When provider returns 429, AI SDK retries by default (up to 3x), tripling quota usage before the error surfaces.
**Why it happens:** `streamText` `maxRetries` defaults to 2.
**How to avoid:** Set `maxRetries: 0` for quota-sensitive endpoints.

### Pitfall 3: BACKLOG-29 Already Implemented
**What goes wrong:** Spending time reimplementing `SeasonsSection` which already exists in `movie-detail-modal.tsx`.
**Why it happens:** Research wasn't done before planning.
**How to avoid:** Verify the season breakdown displays in the TV modal before writing any tasks.

### Pitfall 4: Schema Migration for Settings Name
**What goes wrong:** Adding `full_name` column to profiles table unnecessarily.
**Why it happens:** Assuming DB storage is needed for Google name.
**How to avoid:** Read `user.user_metadata.full_name` directly from Supabase Auth — no DB column needed.

### Pitfall 5: SeriesContent onMovieClick vs Link Navigation
**What goes wrong:** Re-adding `SeriesContent` rows that use modal `onMovieClick`, but Phase 7 established link-based navigation for TV shows (`/tv/[id]`).
**Why it happens:** Copying old component without checking Phase 7 decisions.
**How to avoid:** Pass `hrefPrefix="/tv/"` to `MovieRow` cards instead of `onMovieClick`, or verify that modal approach is still acceptable for the curated rows.

### Pitfall 6: Grid Padding Breaking Infinite Scroll
**What goes wrong:** Padding null items into the array before all pages load causes premature incomplete-row fixes.
**Why it happens:** Padding applied regardless of `hasNextPage`.
**How to avoid:** Only pad when `!hasNextPage && !isFetchingNextPage`.
