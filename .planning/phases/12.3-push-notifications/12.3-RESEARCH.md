# Phase 12.3: Push Notifications - Research

**Researched:** 2026-03-07
**Domain:** Web Push API, Vercel Cron Jobs, Service Worker push events
**Confidence:** HIGH

## Summary

This phase adds per-movie release notification opt-in via the Web Push API. Users tap a bell icon on unreleased movie detail pages to subscribe. A daily Vercel Cron job queries the `tmdb_media` cache for movies releasing today, cross-references with user bell subscriptions, and delivers Web Push notifications via the `web-push` npm package. The service worker (already present via Serwist) handles push event display and notification click navigation.

The architecture is straightforward: two new database tables, two new API routes, a bell icon component on the movie detail page action bar, and push/notificationclick event listeners appended to the existing service worker. All building blocks already exist in the codebase -- Drizzle schema, Serwist service worker, TMDB cache with `release_date` field, and the fixed action bar pattern on detail pages.

**Primary recommendation:** Use `web-push` (server) + native Web Push API (browser) + Serwist service worker extension. No third-party notification service needed.

### Pattern 2: Browser Push Subscription Flow
**What:** Client-side subscription management via PushManager API
**When to use:** When user taps bell icon for the first time (or any time subscription is needed)
**Example:**
```typescript
// Source: MDN Web Push API docs
async function subscribeToPush(): Promise<PushSubscription | null> {
  const registration = await navigator.serviceWorker.ready;

  // Check current permission
  const permission = Notification.permission;
  if (permission === "denied") {
    // Show toast: "Notifications blocked. Enable in browser settings."
    return null;
  }

  if (permission === "default") {
    const result = await Notification.requestPermission();
    if (result !== "granted") return null;
  }

  // Subscribe with VAPID public key
  const subscription = await registration.pushManager.subscribe({
    userVisibleOnly: true,
    applicationServerKey: process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY,
  });

  return subscription;
}
```

### Pattern 3: Service Worker Push Event Handler
**What:** Handle incoming push messages and display OS notifications
**When to use:** In `app/sw.ts`, appended before `serwist.addEventListeners()`
**Example:**
```typescript
// Source: MDN ServiceWorkerGlobalScope push event
self.addEventListener("push", (event) => {
  const data = event.data?.json() ?? {};
  const title = data.title ?? "Moodflix";
  const options: NotificationOptions = {
    body: data.body ?? "",
    icon: "/icons/icon-192x192.png",
    badge: "/icons/icon-72x72.png",
    data: { url: data.url ?? "/" },
  };

  event.waitUntil(
    self.registration.showNotification(title, options)
  );
});

self.addEventListener("notificationclick", (event) => {
  event.notification.close();
  const url = event.notification.data?.url ?? "/";

  event.waitUntil(
    clients.matchAll({ type: "window" }).then((clientList) => {
      // Focus existing tab if already open
      for (const client of clientList) {
        if (client.url.includes(url) && "focus" in client) {
          return client.focus();
        }
      }
      // Otherwise open new tab
      return clients.openWindow(url);
    })
  );
});
```

export async function GET(request: NextRequest) {

  // 2. Query notification_subscriptions JOIN tmdb_media WHERE release_date = today
  // 3. For each match, get all push_subscriptions for that user

  // 5. Handle failed deliveries (remove stale subscriptions)

### Pattern 5: Cron Job Database Query
**What:** Join notification subscriptions with tmdb_media cache to find today's releases
**When to use:** Inside the cron route handler
**Example:**
```typescript
// Query: find all bell-subscribed movies releasing today
const today = new Date().toISOString().slice(0, 10); // "2026-03-07"

const releasingToday = await db
  .select({
    userId: notificationSubscriptions.userId,
    tmdbId: notificationSubscriptions.tmdbId,
    title: tmdbMedia.title,
    posterPath: tmdbMedia.posterPath,
  })
  .from(notificationSubscriptions)
  .innerJoin(
    tmdbMedia,
    and(
      eq(tmdbMedia.tmdbId, notificationSubscriptions.tmdbId),
      eq(tmdbMedia.mediaType, "movie")
    )
  )
  .where(eq(tmdbMedia.releaseDate, today));
```

### Anti-Patterns to Avoid
- **Storing VAPID keys in database:** Use environment variables. Keys are deployment-scoped, not user-scoped.
- **Sending push synchronously in a loop:** Use `Promise.allSettled()` for batch delivery to avoid one failure blocking others.
- **Not cleaning up failed subscriptions:** Push endpoints expire. On 404/410 from sendNotification, delete the subscription row.
- **Using notification icon URL from external domain:** Notification icons must be same-origin or CORS-enabled. Use local icon paths, not TMDB URLs for the notification icon. TMDB poster can be used for the notification `image` property (larger image, optional).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Push payload encryption | Custom RFC 8291 implementation | `web-push` library | Complex cryptography with p256dh/auth key exchange |
| VAPID key generation | Manual key pair creation | `npx web-push generate-vapid-keys` CLI | URL-safe Base64 encoding, proper curve parameters |
| Subscription endpoint management | Custom push service client | `web-push.sendNotification()` | Handles FCM/Mozilla push service differences transparently |
| Permission state machine | Custom permission tracking | Browser `Notification.permission` API | Browser manages grant/deny/default states natively |

**Key insight:** The Web Push protocol involves three parties (browser, push service, app server) with specific encryption requirements. The `web-push` library abstracts all of this into a single `sendNotification()` call.

## Common Pitfalls

### Pitfall 1: VAPID Public Key Format for PushManager.subscribe()
**What goes wrong:** `applicationServerKey` expects a `Uint8Array`, not a raw Base64 string
**Why it happens:** The key from `web-push.generateVAPIDKeys()` is URL-safe Base64, but `subscribe()` needs raw bytes
**How to avoid:** Convert with `urlBase64ToUint8Array()` helper function:
```typescript
function urlBase64ToUint8Array(base64String: string): Uint8Array {
  const padding = "=".repeat((4 - (base64String.length % 4)) % 4);
  const base64 = (base64String + padding).replace(/-/g, "+").replace(/_/g, "/");
  const rawData = atob(base64);
  return Uint8Array.from(rawData, (char) => char.charCodeAt(0));
}
```
**Warning signs:** `InvalidAccessError` or `DOMException` when calling `pushManager.subscribe()`

### Pitfall 2: Stale Push Subscriptions (410 Gone)
**What goes wrong:** Push service returns 410 for expired/unsubscribed endpoints, causing unhandled errors
**Why it happens:** Users clear browser data, uninstall PWA, or browser revokes subscription
**How to avoid:** Catch 410/404 status codes from `webpush.sendNotification()` and delete the subscription row
```typescript
try {
  await webpush.sendNotification(subscription, payload);
} catch (error) {
  if (error instanceof WebPushError && (error.statusCode === 410 || error.statusCode === 404)) {
    await db.delete(pushSubscriptions).where(eq(pushSubscriptions.endpoint, subscription.endpoint));
  }
}
```
**Warning signs:** Growing count of failed deliveries in cron logs

### Pitfall 3: Vercel Hobby Cron Timing Imprecision
**What goes wrong:** Cron configured for `0 8 * * *` may fire anytime between 8:00 and 8:59 UTC on Hobby plan
**Why it happens:** Vercel distributes Hobby cron invocations across the hour for load balancing
**How to avoid:** This is acceptable for daily release notifications (exact minute doesn't matter). If precise timing is needed, upgrade to Pro plan. Document this behavior.
**Warning signs:** Notifications arriving up to 59 minutes later than expected

### Pitfall 4: Service Worker Update and Push Registration
**What goes wrong:** Updated service worker doesn't receive push events until activated
**Why it happens:** Serwist uses `skipWaiting: true` and `clientsClaim: true`, so this is mitigated
**How to avoid:** Already handled -- current Serwist config has `skipWaiting: true` and `clientsClaim: true`, meaning new service worker versions activate immediately
**Warning signs:** Users not receiving notifications after app update

### Pitfall 5: Multiple Tabs and Notification Deduplication
**What goes wrong:** User has multiple tabs open, notification click opens duplicate tab
**Why it happens:** `notificationclick` handler doesn't check existing tabs properly
**How to avoid:** Use `clients.matchAll({ type: "window" })` to find and focus existing tab before opening a new one (shown in Pattern 3 above)
**Warning signs:** Multiple tabs opening to the same movie detail page

### Pitfall 6: Cron Idempotency
**What goes wrong:** Vercel may deliver the same cron event more than once, sending duplicate notifications
**Why it happens:** Vercel's event-driven system can occasionally double-deliver
**How to avoid:** Track sent notifications with a `last_notified_at` timestamp on the `notification_subscriptions` row. Skip if already notified today.
**Warning signs:** Users receiving duplicate notifications for the same movie

### Pitfall 7: tmdb_media.releaseDate May Be Missing
**What goes wrong:** Cron finds no matches because upcoming movies haven't been cached yet
**Why it happens:** `tmdb_media` only caches movies that users have visited (detail page trigger)
**How to avoid:** The cron should also check `tmdb_cache` (upcoming category) or make a direct TMDB API call for movies in `notification_subscriptions` that lack a cached `release_date`. Alternatively, cache the release date when the user subscribes the bell.
**Warning signs:** Zero notifications sent despite movies releasing

export const pushSubscriptions = pgTable("push_subscriptions", {
  id: uuid("id").primaryKey().defaultRandom(),
  userId: uuid("user_id")
    .notNull()
    .references(() => profiles.id, { onDelete: "cascade" }),

    id: uuid("id").primaryKey().defaultRandom(),
    userId: uuid("user_id")
      .notNull()

## Discretionary Recommendations

### Notification Grouping (Claude's Discretion)
**Recommendation:** Send one notification per movie (not grouped). Multiple movies releasing on the same day is rare, and individual notifications are more actionable (each opens a specific movie page). If 3+ movies release same day, still send individually -- users opted into each explicitly.

### Bell Icon Placement (Claude's Discretion)
**Recommendation:** Place in the fixed bottom action bar, after the like/dislike buttons and before the Trailer link. Follows the existing icon layout pattern. Bell only appears for unreleased movies (where `release_date > today`).

### Stale Subscription Cleanup (Claude's Discretion)
**Recommendation:** Clean up on failed delivery (reactive). When `sendNotification()` returns 404/410, delete the `push_subscriptions` row immediately. No periodic cleanup job needed -- the daily cron naturally prunes stale subscriptions as it encounters them.

### VAPID Key Generation (Claude's Discretion)
**Recommendation:** Generate once via `npx web-push generate-vapid-keys`, store as `NEXT_PUBLIC_VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, and `VAPID_SUBJECT` environment variables in `.env.local` and Vercel dashboard. Public key needs `NEXT_PUBLIC_` prefix since the browser needs it for `pushManager.subscribe()`.

### Bell Icon Visual Design (Claude's Discretion)
**Recommendation:** Use `Bell` (outline) for unsubscribed, `BellRing` (filled via `fill-current`) for subscribed. Matches the existing bookmark icon pattern (outline vs filled). Use crimson accent color when subscribed (`text-primary fill-primary`).

## Open Questions

1. **TMDB release_date reliability**
   - What we know: TMDB stores `release_date` as a string (e.g., "2026-04-15"). The `tmdb_media` table caches this.
   - What's unclear: TMDB release dates can change. If a movie's release date shifts after a user subscribes, the cached date in `notification_subscriptions` may be stale.
   - Recommendation: Cache `release_date` at bell-subscribe time, but the cron should match against the `tmdb_media` cache (which refreshes on detail page visits). If no `tmdb_media` row exists for a subscribed movie, the cron should fetch fresh data from TMDB for that specific movie.

2. **Notification icon assets**
   - What we know: Notification `icon` should be a local path (same-origin). PWA icons already exist at `/icons/`.
   - What's unclear: Whether existing PWA icons are the right sizes for notification badge/icon.
   - Recommendation: Reuse existing `/icons/icon-192x192.png` for notification icon and `/icons/icon-72x72.png` for badge. Verify these paths exist in the current PWA setup.

3. **Vercel Hobby plan: one cron job is free?**
   - What we know: Hobby allows 100 cron jobs but restricted to once-per-day minimum interval with hourly precision.
   - What's unclear: Whether the project is on Hobby or Pro plan.
   - Recommendation: Daily at 8 AM UTC works on both plans. On Hobby, actual trigger may be 8:00-8:59 UTC -- acceptable for release notifications.
